# Knowledge Governance Console 知识治理控制台

这份文档用于说明“知识治理控制台”作为作品集新增项目时的定位、功能、架构边界和能力证明。该项目来自本地 Codex 会话 `019e3b51-b4a9-74f1-b888-820c6c1f7895`，是一个独立运行的 Streamlit 小工具，不嵌入 AI 运营分身 Agent 代码。

## 项目定位

知识治理控制台用于解决 Agent 知识库长期维护中的一个实际问题：作品集、运营方法论和业务规则不断增加后，不能只依赖手工改文档和重新构建索引，还需要一个可以登记、审计、追溯、修正和同步导出的治理入口。

它不是另一个聊天 Agent，而是一个知识管理后台。控制台负责把知识单元结构化保存到本地 SQLite 数据库，再把生效知识导出为 Agent 可读取的 YAML 文件。Agent 端只需要在索引构建时选择是否扫描控制台的 `sync/` 导出目录，就能把这些知识纳入本地知识库。

## 核心功能

1. 知识单元注册中心：录入标题、内容、类型、创建者、负责人、审核人、生效日期、失效日期、来源、可信度、版本、适用范围和状态。
2. 知识库健康度仪表盘：展示知识数量、状态分布、待验证告警和同步文件统计。
3. 知识清单筛选：按状态、类型、负责人等维度查看和筛选知识单元。
4. 责任归属与错误追溯：为每条知识保留 owner、approver、source、confidence、version 等字段，便于定位问题来源。
5. Badcase 回流与审核看板：记录错误案例、修正建议和验证状态，让错误回答可以变成可治理的知识资产。
6. YAML 同步转译：将“生效中且未过期”的知识单元导出到 `sync/<kid>.yaml`，供下游 Agent 索引脚本读取。

## 架构边界

控制台采用本地 Streamlit + SQLite + YAML 导出结构，保持与 AI 运营分身 Agent 的低耦合关系。

- 控制台不调用 ChromaDB。
- 控制台不修改 Agent 代码。
- 控制台不依赖 Agent 内部模块。
- Agent 独立运行时仍只扫描自己的 `works/` 文件夹。
- 控制台同步路径默认关闭，只有用户显式配置后才会被 Agent 索引脚本读取。

这种边界设计让两个系统可以各自独立运行：控制台负责知识治理，Agent 负责问答、检索和展示。二者通过 `sync/` 文件夹和 YAML 格式衔接。

## 与 AI 运营分身 Agent 的关系

AI 运营分身 Agent 解决的是“如何把作品集和运营工作流做成可交互系统”的问题；知识治理控制台解决的是“当 Agent 的知识越来越多时，如何管理知识质量、责任归属和修正流程”的问题。

两者形成一条更完整的工作闭环：

1. 用户在 Agent 中提问，暴露知识缺口或错误回答。
2. 运营者在控制台中登记 Badcase，并补充或修正知识单元。
3. 控制台将生效知识导出为 YAML。
4. Agent 索引脚本在用户启用同步路径后读取 `sync/`，把修正后的知识纳入检索。
5. 下一次问答时，Agent 可以基于更新后的知识库回答。

## 证明能力

这个项目证明的不只是“又做了一个后台页面”，而是我开始把 AI Agent 当作需要长期治理的系统来看待：

- 知识治理意识：知道 RAG 不是一次性塞文档，而需要注册、审核、追溯、修正和失效管理。
- Agent 产品化思维：把错误回答、知识变更和同步索引纳入真实运维流程，而不是只关注对话生成。
- 低耦合架构意识：通过 YAML 和配置开关连接控制台与 Agent，避免让两个项目互相硬依赖。
- 交付边界意识：控制台默认独立运行，Agent 默认行为不变，需要接入时再显式启用同步路径。
- 数据结构化能力：用 kid、status、confidence、scope、version、owner 等字段把散乱知识变成可管理资产。

## 面试表达建议

如果面试官问“这个知识治理控制台和 AI 运营分身有什么关系”，可以这样回答：

我一开始做 AI 运营分身时，重点是让作品集能被检索和对话。但继续迭代后，我意识到 Agent 的知识库不是静态材料库，而是会持续新增、修正和淘汰的资产。所以我又做了一个独立的知识治理控制台，用来登记知识单元、追踪 Badcase、管理审核状态，并把生效知识导出为 YAML。Agent 端默认不依赖它，但可以通过配置启用 `sync/` 路径，把控制台中修正过的知识纳入索引。这说明我不只会做一个能演示的 Agent，也会考虑它后续如何维护、纠错和治理。
