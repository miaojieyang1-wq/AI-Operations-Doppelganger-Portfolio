# AI运营分身 Agent 设计与迭代记录

## 文档定位

这份文档用于说明“AI运营分身作品集网站”本身的设计思路、迭代路径和能力证明。它不是普通网页说明，而是这个项目作为作品集核心作品时的完整项目日志。

当 HR 或使用者询问以下问题时，应优先参考本文档：
- 这个 AI 分身是怎么设计的？
- 这个 Agent 和普通聊天机器人有什么区别？
- 这个作品集网站经历了哪些版本迭代？
- RAG、Prompt 按需加载、LangGraph 分别解决了什么问题？
- 这个项目为什么能作为作品集核心作品？
- 这个项目体现了哪些运营、产品、AI 工程和数据分析能力？

回答时应使用第一人称“我”，把这个项目表述为“我搭建的 AI 运营分身作品集网站”。

## 总体设计思路

这个 Agent 的目标不是做一个泛泛聊天机器人，而是把我的简历、作品集、SEO Prompt 工程、已发布文章、运营判断能力和工作流工具整合成一个可交互的 AI 作品集。

它面向的不是技术用户，而是 HR、面试官和想快速了解我能力的人。因此，项目设计始终围绕三个问题：

1. 如何让访问者一打开网站就知道该做什么？
2. 如何让我的经历、作品和能力不是静态堆叠，而是能被提问、检索和解释？
3. 如何让这个项目本身证明我具备 AI 产品化、运营思考、数据敏感度和迭代能力？

因此，这个项目逐步从“一个会回答问题的网页”迭代成“面向运营生命周期的 AI 工作台”：

打开页面后，用户可以先通过决策看板进入不同工作场景，也可以在“关于我”里进行模拟面试、查看作品集原文、生成自我诊断，还可以在用户洞察、版本决策、活动策划等模式中完成简单运营工作辅助。

## 当前系统架构

当前版本的系统由五层组成：

1. 前端体验层  
   使用 Streamlit 搭建页面，包括入口弹窗、侧边栏、决策看板、模式切换、报告展示、图表展示和运行状态折叠面板。

2. 模式工作台层  
   包含“决策看板”“关于我”“版本决策”“用户洞察”“活动策划”等模式。

3. Agent 流程编排层  
   使用 LangGraph 实现“意图识别 → 检索增强 → 生成回复”的处理流程。

4. Prompt 按需加载层  
   将原来的长 Prompt 拆成多个小模块，根据用户问题只加载必要内容，降低模型注意力稀释。

5. 本地知识库层  
   读取 works/ 文件夹中的作品文档，构建本地轻量向量库，让 Agent 在作品集、提示词工程、文章案例等问题上可以检索真实材料。

## V1：基础 AI 运营分身

### 设计目标

V1 的目标是先让网站具备最基本的“可问答”能力。这个阶段的重点不是复杂架构，而是验证：用户能不能通过网页快速了解我。

### 核心设计

V1 使用 Streamlit 搭建基础界面，调用 DeepSeek API 生成回答，并通过 `prompt_background.txt` 写入我的长期背景、实习经历、项目经历、游戏理解和回答规则。

这一版主要依赖系统提示词回答：
- 我做过什么实习？
- 我怎么用 AI 做运营？
- 我对哪款游戏理解最深？
- 我策划过什么活动？

### 解决的问题

V1 解决了“从静态简历到可对话展示”的问题。用户不需要翻文档，而是可以直接向 AI 分身提问。

### 暴露的问题

- 过度依赖长 Prompt，容易被旧信息干扰。
- 作品集材料没有真正接入，只能靠摘要描述。
- 用户问具体文档内容时，模型不一定能准确引用。
- 不同场景共用一套提示词，回答容易混用语气。

### 能力证明

V1 证明我能把个人经历转化成 AI 可理解的长期背景，并通过 Prompt 设计让 AI 以第一人称进行面试式表达。

## V2：界面产品化与运营视角优化

### 设计目标

V2 的目标是让这个工具不只是“能用”，而是让 HR 打开后马上知道该做什么。

### 核心设计

这一版重点优化前端交互：
- 顶部加入欢迎语和网站定位。
- 侧边栏用大按钮区分模式。
- 当前模式高亮，减少用户迷路。
- 常见问题改为折叠提示，降低视觉干扰。
- 增加入场弹窗，说明网站架构、灵感来源和使用方式。
- 将运行状态和技术日志默认隐藏，避免界面过于开发者化。
- 将聊天记录改为类似即时通讯的气泡展示，并限制聊天区域高度，避免页面无限变长。

### 解决的问题

V2 解决了“用户打开后不知道点哪里”的问题。界面从开发工具感转向作品集展示感，更符合 HR 和非技术用户的使用场景。

### 暴露的问题

界面虽然更清晰，但底层仍主要依赖大 Prompt。作品集内容、工作能力工具和个人问答之间还没有形成更稳定的分工。

### 能力证明

V2 证明我不仅关注 AI 输出，也关注用户路径、信息层级、按钮文案、状态反馈和使用者心理。这体现的是运营视角下的产品体验优化能力。

## V3：RAG 本地知识库接入

### 设计目标

V3 的目标是让 Agent 不再只依赖系统提示词，而是能真正检索我的作品集文档。

### 核心设计

V3 增加本地知识库能力：
- 读取 `works/` 文件夹下的作品文档。
- 支持 Markdown 和 Word 文档。
- 将文档切分成小文本块。
- 构建本地轻量向量库。
- 用户询问作品集、提示词工程、SEO 文章时，优先检索相关材料。

技术方案经历过多轮取舍。项目曾尝试 Chroma、sentence-transformers、fastembed 等方案，但在 Windows 本地环境中遇到 torch DLL、Rust 依赖、Chroma 后端等稳定性问题。最终采用本地轻量向量方案，将向量数据保存为 `chroma_db/local_vectors.json`，避免外部 Embedding API 和复杂依赖。

### 解决的问题

V3 解决了“作品集内容无法被 Agent 准确调用”的问题。用户问提示词工程、迭代记录、文章案例时，Agent 可以检索本地知识库，而不是只凭提示词概括。

### 暴露的问题

轻量向量方案稳定，但语义能力弱于专业 embedding。为了避免不相关文档污染回答，系统加入了相关度阈值和检索范围控制。

### 能力证明

V3 证明我具备 RAG 知识库组织能力，也体现了工程取舍能力：在依赖不稳定时，不强行追求复杂方案，而是选择能稳定运行、能服务展示目标的本地方案。

## V4：LangGraph 流程编排与 Agent 化

### 设计目标

V4 的目标是让系统从“检索增强问答”升级为“有流程的 Agent”。它不再是用户问什么就直接回答，而是先判断意图，再决定是否检索，再选择不同提示词生成回复。

### 核心设计

V4 引入 LangGraph，形成固定流程：

意图识别 → 检索增强 → 生成回复

关键节点：

1. `node_classify_intent`  
   判断用户问题属于 chat、works 还是 analysis。

2. `node_retrieve_context`  
   只在作品集、提示词工程等需要材料支撑的问题上检索知识库；工具型任务不检索作品集，避免无关材料干扰。

3. `node_generate_response`  
   根据意图选择系统提示词并调用 DeepSeek。

### 解决的问题

V4 解决了“所有问题都用同一套回答方式”的问题。它让 Agent 具备基础流程判断能力，能根据问题类型切换角色和输出结构。

### 进一步优化

V4 还加入了多轮上下文。了解我模式保留最近 8 轮问答，让用户可以追问；工作报告类任务保持单轮，避免不同报告互相污染。

### 能力证明

V4 证明我能把 AI 应用从单轮问答推进到流程化 Agent：有意图识别、有检索策略、有场景化提示词、有错误兜底、有运行状态反馈，也有多轮上下文管理。

## V5：决策看板与运营生命周期工作台

### 设计目标

V5 的目标是把工具从“几个分散功能入口”升级成“运营决策中心”。用户不需要先理解每个功能叫什么，而是可以按产品生命周期选择当前要做的运营工作。

### 核心设计

新增默认首页“运营决策中心”，按时间线排列五个阶段：
- 研发期：用户需求洞察与市场验证。
- 测试期：版本质量评估与反馈清洗。
- 上线前：卖点提炼、宣发策略与活动方案制定。
- 上线后：版本表现监控与竞品动态追踪。
- 长线运营：持续优化迭代与活动运营。

每个阶段卡片都提供跳转按钮，例如：
- 进入用户洞察
- 竞品分析
- 反馈清洗工作台
- 版本分析
- 活动策划

### 解决的问题

V5 解决了“功能很多但用户不知道该用哪个”的问题。看板把工具放进真实运营流程里，让使用者从任务场景进入，而不是从技术模块进入。

### 进一步优化

为了支持从决策看板跳转到指定子功能，用户洞察和版本决策的顶部 Tab 被改造成可被状态控制的横向选择控件。这样按钮点击后能准确定位到“访谈助手”“反馈清洗”“版本分析”“竞品雷达”等具体工作台。

### 能力证明

V5 体现的是运营产品化思维：我不是简单堆功能，而是把功能重新组织成研发期到长线运营的工作流。

## V6：Prompt 分类归档与按需加载

### 设计目标

V6 的目标是解决长 Prompt 带来的注意力稀释问题。此前完整背景文档很长，如果每次调用都整段塞给模型，容易导致模型抓错重点。

### 核心设计

新增 `prompts/` 分类目录和 `prompt_loader.py`，将 Prompt 拆成小模块：

- `shared/base_identity.txt`：基础身份和表达规则。
- `about/profile_core.txt`：基础背景和数据分析表达规则。
- `about/internships.txt`：实习经历。
- `about/projects.txt`：项目与活动经历。
- `about/game_understanding.txt`：游戏理解。
- `works/portfolio.txt`：作品集回答规则。
- `works/prompt_engineering.txt`：提示词工程回答规则。
- `tasks/version_analysis.txt`：版本分析规则。
- `tasks/diagnosis.txt`：自我诊断规则。

系统根据用户问题选择最小必要 Prompt：
- 问实习，只加载身份、基础背景、实习模块。
- 问游戏理解，只加载身份和游戏理解模块。
- 问提示词工程，只加载作品集、提示词工程和必要检索内容。
- 做版本分析，只加载版本分析规则，不加载个人简历。

### 解决的问题

V6 解决了“回答被无关背景带偏”的问题。比如问《崩坏：星穹铁道》时，不再让模型同时阅读大量 SEO 工程背景；问版本公告分析时，也不会把个人经历塞进工作报告。

### 自检结果

改造后，典型调用的 Prompt 字符量明显下降：
- 实习问题约为原完整背景的 10%。
- 游戏理解问题约为原完整背景的 5%。
- 版本分析约为原完整背景的 5%。

### 能力证明

V6 证明我理解 Prompt 工程不只是“写得越全越好”，而是要控制模型注意力，把上下文设计成可路由、可组合、可检查的系统。

## V7：数据敏感度与运营分析框架强化

### 设计目标

V7 的目标是让 Agent 输出不只是“会总结”，而是更像运营同学：能说明判断依据、能建立定量分析框架、能体现逻辑推理。

### 核心设计

这一版主要强化系统提示词和对话规则：

1. 版本决策  
   在“内容效果预判”中要求每个判断附带推理逻辑：
   - 爆点预测要说明基于哪类玩家的什么行为规律。
   - 风险预警要说明参考哪个历史版本的数据趋势，若没有数据则明确需要补充验证。
   - 传播建议要说明基于哪个平台的传播规律选择钩子类型。

2. 问卷工坊  
   在问卷末尾加入“量化分析框架”，要求说明：
   - 可从哪 3-4 个维度分析。
   - 每个维度由哪些题目量化。
   - 哪些维度可以交叉验证假设。

3. 反馈清洗  
   将原来的简单情感极性升级为多维情绪分析：
   - 基础情绪
   - 情绪强度
   - 情绪对象
   - 复合情绪
   - 发言动机标签

4. 了解我  
   当用户问数据分析能力时，不让 Agent 只列 Excel、SQL，而是要求用具体案例表达“拆解 → 假设 → 验证”的思维过程。

### 解决的问题

V7 解决了“输出像普通 AI 总结”的问题。它让报告更强调依据、比例、频次、交叉分析和风险判断，而不是只给主观形容词。

### 能力证明

V7 证明我在运营分析中关注数据敏感度和逻辑链条。即使没有真实后台数据，也会主动标注需要验证的数据口径，而不是假装确定。

## V8：反馈可视化、验收测试与交付稳定性

### 设计目标

V8 的目标是提升工具的交付完整度：不仅能生成文字报告，还能把反馈清洗结果转化为可视化图表，并通过系统验收降低现场展示风险。

### 核心设计

这一版做了几类优化：

1. 反馈清洗可视化  
   在反馈清洗报告下方自动生成两张图：
   - 基础情绪分布饼图。
   - 发言动机分布条形图。

2. 图表兼容性修复  
   由于 matplotlib 默认字体不支持中文，图表曾出现空心方框。后续通过加载 Windows 中文字体（微软雅黑、黑体、宋体）修复中文显示问题。

3. 注意力检查与 Prompt 压缩  
   对所有 Prompt 做长度检查，发现 `feedback_clean_prompt.txt` 规则密度较高，因此先备份再压缩为表格化规则，将长度从 2533 字符、118 行压缩到 1770 字符、51 行，同时保留核心判断标准。

4. RAG 检索片段控制  
   将作品集检索片段数量从 5 降到 3，减少检索上下文过多导致的回答跑偏。

5. 完整验收清单  
   对项目做语法、依赖、模式跳转、知识库、DeepSeek 配置、UI、输出质量等维度的验收，明确哪些通过、哪些需要真实 API 环境继续测试。

### 解决的问题

V8 解决了“作品集现场展示不稳定”的问题。它不只关注功能是否存在，还关注：
- 图表是否能正确显示中文。
- Prompt 是否过长导致误读。
- 页面是否能启动。
- 作品集文档是否可打开。
- 知识库是否可读取。
- 错误提示是否对非技术用户友好。

### 能力证明

V8 证明我具备面向交付的自检意识。一个 AI 作品集不是“能跑一次就算完成”，而是要考虑依赖、字体、异常兜底、用户路径和现场展示风险。

## V9：作品集文档预览与面试官阅读体验优化

### 设计目标

V9 的目标是让“我的作品集”不只是能展示文件内容，而是更接近真实文档阅读场景。此前原文内容被放在页面底部或文本框中，用户点击后不容易发现内容位置，也丢失了 Word 文档原本的标题层级、大纲和跳转感。

### 核心设计

这一版做了几类优化：

1. 原文预览位置前移  
   将作品原文从页面底部统一展示，改为在对应作品卡片所在行下方全宽展开。这样用户点击“预览文档”后，能立刻在当前上下文中看到内容。

2. 文档预览与原文件打开分离  
   网页内提供轻量预览，同时增加“打开原文件”和“下载原文件”入口。这样网页负责快速扫读，Word 文件负责保留原始大纲、目录、跳转和排版。

3. 文档大纲提取  
   对 Markdown 和 docx 文档进行标题层级识别，在预览区域上方展示“查看文档大纲”，帮助 HR 快速判断文档结构。

4. 纯文本预览  
   将网页内正文预览改为纯文本显示，避免 Markdown 样式干扰作品本体阅读，也避免让用户误以为源文件内容被改动。

5. 作品集引导提示  
   在作品集顶部增加提示，引导访问者如果对作品有疑问、想按岗位能力拆解，或希望进一步解释设计思路，可以切回“了解我”与 AI 运营分身对话。

### 解决的问题

V9 解决了“作品集原文能打开，但阅读体验不像作品集”的问题。它让文档展示更符合面试官的浏览习惯：先看卡片摘要，再看大纲，再决定是否打开原文件深读。

### 能力证明

V9 证明我不仅关注功能是否存在，也关注材料如何被目标读者理解。作品集不是文件堆叠，而是需要被设计成一种清晰的阅读路径。

## V10：演示模式与数据源真实性预警机制

### 设计目标

V10 的目标是提高作品集现场演示的稳定性和分析输出的可信边界。此前系统依赖实时 API，一旦网络、额度或接口状态不稳定，现场展示就可能中断；同时，AI 报告中的推理判断也需要清楚标注数据来源，避免被误读为真实后台数据。

### 核心设计

这一版新增两套机制：

1. 演示模式  
   在侧边栏加入“演示模式”开关。开启后，所有生成结果从本地 `demo_responses.json` 读取，不实时调用外部 API。  
   这让面试、路演、无网环境或 API 不稳定时仍能稳定展示完整流程。

2. 本地演示响应引擎  
   新增 `demo_engine.py`，根据当前任务文本匹配不同演示结果，例如：
   - 个人介绍
   - 版本分析
   - 竞品雷达
   - 用户洞察
   - 活动策划
   - 反馈清洗
   - 多职能协作讨论

3. 数据源真实性预警  
   在页面顶部加入“数据源真实性预警”开关，默认开启。开启后，所有分析报告顶部都会显示黄色预警横幅，提醒使用者报告基于用户输入和 AI 推理生成，未经过实时数据校验。

4. 分模块风险提示  
   根据不同功能追加不同风险提示。例如：
   - 竞品雷达提示未接入实时竞品数据源。
   - 版本分析提示效果预判可能受市场环境影响。
   - 反馈清洗提示结论取决于样本代表性。
   - 活动策划提示数值设计需灰度测试。

5. 数据源声明  
   在所有分析类提示词中追加“数据源声明”要求，并在保存/导出时做兜底补充。即使用户关闭顶部预警，报告末尾仍会保留数据源声明。

### 解决的问题

V10 解决了两个关键问题：

- 现场演示稳定性：不再完全依赖实时 API。
- AI 输出边界：明确哪些判断来自用户输入，哪些属于 AI 推理，哪些需要官方数据、用户调研或 A/B 测试验证。

### 能力证明

V10 证明我具备产品化的风险意识。一个可交付的 AI 工具不仅要能生成内容，也要考虑离线兜底、展示稳定性、数据来源透明度和决策责任边界。

## V11：多职能协作讨论模式

### 设计目标

V11 的目标是让 Agent 不只是“单一助手回答问题”，而是模拟真实游戏运营团队中的多职能讨论。版本决策往往不是一个人完成的，而是市场、用户、社区、产品、活动、渠道、数据、内容、版本、商业化等角色共同权衡的结果。

### 核心设计

这一版在“版本决策”中新增“多职能协作讨论模式”。

1. 十个独立职能角色  
   新增十个角色系统提示词文件：
   - 市场运营
   - 用户运营
   - 社区运营
   - 产品运营
   - 活动运营
   - 渠道运营
   - 数据运营
   - 内容运营
   - 版本运营
   - 商业化运营

2. 角色边界规则  
   每个角色都有明确回答边界：如果版本公告缺乏该职能相关信息，不能强行编造，而是输出“视角建议”。这避免了多角色系统为了凑观点而产生空泛或虚假的分析。

3. 并发独立调用  
   每个被选中的角色对应一次独立 API 调用。单个角色失败不会影响其他角色，超过半数失败时才停止生成综合建议。

4. 运营目标导向  
   用户可以选择本次协作目标，例如：
   - 用户留存
   - 危机公关
   - 新用户获取
   - 回流用户召回
   - 商业化收入提升
   - 品牌口碑建设
   - 社区活跃度提升
   - 版本质量保障
   - 用户自定义

5. 协调者综合建议  
   所有角色输出完成后，由“运营总监”协调者读取各职能观点，根据运营目标区分直接相关、间接相关和补充视角，最终输出目标导向的行动方案。

6. 可折叠结果展示  
   各职能观点用折叠面板展示。未选择的角色不显示；输出“视角建议”的角色会在标题后追加标签，并用虚线样式提示其不是正式分析意见。

7. 报告保存  
   协作讨论报告保存到 `reports/collaboration/` 文件夹，文件名包含运营目标和时间戳，并纳入数据源真实性预警机制。

### 解决的问题

V11 解决了“单 Agent 视角容易单一”的问题。通过多职能角色拆分，系统可以模拟真实运营会议中的多方权衡：有人关注传播，有人关注留存，有人关注数据，有人关注商业化，有人关注版本稳定性。

### 能力证明

V11 证明我对游戏运营不是只停留在单点任务，而是理解运营决策中的组织协作关系。它体现了我将复杂工作流拆成角色、目标、边界、汇总和交付物的能力。

## V12：运营路线图、首屏速度与结果定位体验优化

### 设计目标

V12 的目标是继续降低使用门槛，让 HR 或使用者打开网站后更快理解“我现在应该从哪里开始”，并减少等待、找不到结果、功能太多不知如何进入的问题。

前几个版本已经搭建出完整工具能力，但随着功能增多，首页如果只是功能卡片，仍然需要使用者自己判断“研发期该用哪个工具”“上线后该看哪个入口”。因此 V12 把默认首页从功能卡片进一步升级为“运营路线图”，让工具入口被组织进真实运营流程。

### 核心设计

这一版主要包含四类优化：

1. 运营路线图首页  
   将原来的五张生命周期卡片改造成从上到下的流程路线图。五个阶段分别是：
   - 研发期：用户需求洞察与市场验证
   - 测试期：版本质量评估与反馈清洗
   - 上线前：卖点提炼、宣发策略与活动方案制定
   - 上线后：版本表现监控与竞品动态追踪
   - 长线运营：持续优化迭代与活动运营

   每个阶段左侧有阶段序号圆圈，阶段之间用竖向虚线箭头连接，页面顶部增加流程总览条。用户展开某个阶段后，对应圆点高亮。

2. 阶段工作台模式  
   每个阶段默认只显示阶段名称和说明，展开后显示：
   - 运营目标
   - 关键产出
   - 纵向排列的功能入口
   - 每个入口对应的用途说明

   例如研发期展开后，会提示该阶段目标是明确目标用户画像、核心卖点和竞品差异化定位；可进入用户洞察或竞品分析。

3. 首屏与弹窗加载优化  
   此前弹窗图片和完整页面脚本一起加载，用户可能感觉网页打开后有等待。V12 做了两层优化：
   - 将弹窗图片压缩为网页专用 WebP，原图保留，弹窗优先加载轻量图。
   - 将 `openai`、`agent_graph` 等重模块改为延迟加载，未点击确认前只加载弹窗专用轻量页面。

   这样首页阶段不再提前加载完整 Agent 流程和外部 API 客户端，弹窗出现速度明显提升。

4. 结果区自动定位  
   多数功能的输出原本在页面下方，用户点击生成后需要手动下滚才能看到结果。V12 增加统一结果锚点：报告生成后刷新页面时自动滚动到结果区，并在结果顶部显示“内容已生成，已为您定位到结果区”的提示。

   已覆盖：
   - 用户洞察：访谈助手、问卷工坊、反馈清洗
   - 版本决策：版本分析、竞品雷达、协作讨论
   - 活动策划
   - 自我诊断
   - 通用报告输出区

5. 演示模式自检  
   对本地 `demo_responses.json` 和 `demo_engine.py` 做了功能性检查，确认演示模式下可以覆盖个人介绍、自我诊断、版本分析、竞品雷达、活动策划、访谈助手、问卷工坊、反馈清洗、多职能协作角色和协调者输出。  
   演示模式可以在不调用外部 API 的情况下稳定返回本地预设结果，适合面试现场展示。

6. Agent 功能介绍 Prompt 修正  
   将“请全面介绍你的所有内容并指引我进行操作”改为“请全面介绍本Agent所有功能并指引我进行操作”，避免模型把问题理解成候选人简历介绍。  
   同时更新 Prompt 路由与作品集 Prompt，使“本 Agent 功能”“如何操作”“运营路线图”“演示模式”“多职能协作”等问题优先进入作品集 / Agent 项目说明路径，而不是个人经历路径。

### 解决的问题

V12 解决了三个使用体验问题：

- 功能很多但入口理解成本高：通过运营路线图，把功能放进真实工作阶段。
- 打开页面或弹窗等待感明显：通过轻量弹窗页面、图片压缩和延迟加载减少首屏负担。
- 生成内容在页面下方不易发现：通过结果锚点和自动定位，让用户生成后直接看到输出。

### 能力证明

V12 证明我不只会继续增加功能，也会在功能变多后重新组织信息架构、优化首屏性能、处理用户注意力路径，并持续校正 Prompt 语义。  
这体现了 AI 产品化中很重要的一点：功能不是越多越好，关键是使用者能不能快速理解、稳定进入、顺利拿到结果。

## V13：框架梳理、LangGraph 化推进与交付稳定性优化

### 迭代背景

V13 的目标是把前面不断新增的功能重新收束到一个更清晰、更稳定的 Agent 框架里。前期系统已经具备 RAG、LangGraph、Prompt 按需加载、多职能协作、演示模式和数据源预警，但部分功能仍然存在“UI 可以用、底层流程不够统一”的问题。

因此，这一版重点不是再增加新入口，而是检查哪些功能还没有被纳入统一流程，哪些模块还容易拖慢加载，哪些交付文件还不够适合给技术朋友或面试官检查。

### 关键改动

1. LangGraph 化推进  
   对尚未纳入 LangGraph 流程的分析类功能做梳理，并把适合统一编排的部分逐步接入任务图。这样系统不只是页面按钮调用 API，而是更接近“任务输入 → Prompt 路由 → 生成 → 保存/展示”的 Agent 工作流。

2. 工作流可视化  
   在侧边栏运行状态区域增加“查看工作流”入口，可以直接展示 `agent_graph.py` 中 LangGraph 的 Mermaid 文本；如果 Mermaid 获取失败，则回退到 ASCII 流程图。这让使用者和技术检查者能看到 Agent 的真实流程结构，而不是只看页面效果。

3. 交付包完善  
   对项目做打包，并补充解压、安装、运行指南。这样技术朋友拿到项目后，不需要从聊天记录里拼信息，可以直接按指南检查代码、安装依赖和启动应用。

4. 加载速度与稳定性优化  
   对首页弹窗、图片加载、首屏初始化和模块导入做检查，减少打开页面时不必要的重型加载。尤其是避免首页阶段提前加载完整 Agent 流程和外部 API 客户端，让作品集网站更适合现场演示。

5. 模块拆分与低风险代码优化  
   对 `app.py` 中已经稳定的部分做低风险拆分和整理，例如把协作角色配置独立到 `collaboration_config.py`，把部分报告格式化、样式和 UI 辅助逻辑迁移到工具模块。目标是降低单文件复杂度，同时不改变现有功能行为。

### 解决的问题

V13 主要解决三个问题：

- 功能越来越多后，底层流程需要更容易解释。
- 演示和交付场景下，页面打开速度和运行稳定性需要更可靠。
- 项目需要能被技术朋友检查，而不是只能由开发者本人解释。

### 能力证明

V13 证明这个 Agent 不只是“功能堆叠”，而是在不断向可维护、可解释、可交付的产品形态靠近。它体现了我对工程交付完整度的理解：功能能跑只是第一步，还需要能说明流程、能被检查、能被打包、能在演示现场稳定呈现。

## V14：结构化交付、协作报告保存与管理后台补完

### 迭代背景

V14 的目标是补上此前设计中遗漏的几个关键闭环：分析报告不仅要能在网页里阅读，也要能被后续系统继续处理；多职能协作讨论不仅要能生成结果，也要能由使用者确认后保存；项目本身也需要一个轻量管理后台，方便查看报告、知识库、配置和缓存状态。

这一版不新增新的业务模式，而是完善已有功能的完整度，让系统更像一个可以持续迭代的运营工具。

### 关键改动

1. JSON 结构化输出  
   在所有分析类系统提示词末尾追加结构化 JSON 输出要求，包括：
   - `report_type`：报告类型
   - `timestamp`：生成时间
   - `key_findings`：关键发现
   - `metrics`：量化指标
   - `risk_level`：风险等级
   - `action_items`：行动建议
   - `data_confidence`：数据置信度说明

   同时在应用保存 Markdown 报告时，自动生成同名 JSON 文件并存入 `reports/json/`。如果 JSON 生成失败，不阻塞 Markdown 报告保存，保证用户主流程不受影响。

2. 协作讨论报告手动保存  
   多职能协作讨论从“生成后自动保存”改为“结果下方点击保存协作报告”。这样更符合用户操作直觉：先看结果，再决定是否保存。

   保存内容包括：
   - 所有参与职能角色的观点
   - 协调者综合建议
   - 本次运营目标

## V15：真实分发安全化、运行时数据隔离与显式持久化

### 迭代背景

V15 来自一次面向“真实可分发产品”的高优先级审查。此前项目已经能作为本地演示作品运行，也能打包成源码包交给技术朋友检查，但如果把它假设成“普通用户可以自行安装使用”的产品，就会暴露出更真实的交付问题：运行时数据写入位置不安全、后台认证不能依赖公开口令、安装链路对 Python 运行时的依赖表达不清、API 接口缺少基础安全边界、报告和问答内容存在默认落盘风险。

这一版的目标不是继续增加功能，而是把项目从“能演示”推进到“更接近可分发、可解释、可控风险”的状态。它解决的是产品交付中的安全边界、隐私边界和运行环境边界问题。

### 关键改动

1. 运行时数据目录隔离

   新增 `runtime_paths.py`，统一管理用户可写数据目录。项目将应用资源目录和运行时数据目录分开：

   - 应用资源目录：代码、配置模板、作品集文档、静态图片等随项目分发的只读资源。
   - 用户数据目录：`.env`、数据库、报告、JSON 附加文件、知识库索引、缓存和后台安全配置。

   默认运行时数据目录遵循系统习惯：

   - Windows：`%LocalAppData%/AI_Ops_Agent`
   - macOS：`~/Library/Application Support/AI_Ops_Agent`
   - Linux：`~/.local/share/AI_Ops_Agent`

   同时支持通过 `AI_OPS_DATA_DIR` 环境变量覆盖。旧版本项目根目录中的 `.env` 和 `local_demo.db` 会被非破坏性复制到新目录，不直接删除旧文件。

2. 后台认证安全化

   新增 `security_utils.py`。后台密码改为首次使用时强制初始化，并只保存带盐哈希。

   具体变化包括：

   - `config/app.yaml` 不再保存明文后台密码。
   - 管理后台未初始化时不会放行，也不会回退到公开口令。
   - 密码使用 PBKDF2-SHA256 加盐哈希保存到用户数据目录的安全配置文件中。
   - 安装向导即使检测到 `.env` 已存在，也不会跳过后台安全初始化。

   这个改动让项目从“方便演示”转向“可以安全分发给别人体验”。

3. 安装与启动链路诚实化

   对 `run.bat`、`run.sh`、`start_app.bat` 和 `setup.iss` 做一致性修正。当前分发方式被明确定位为“源代码分发版”，不再伪装成内置 Python 的独立安装包。

   核心原则是：如果没有真正打包 Python 运行时，就必须明确要求用户预装 Python 3.10+。

   因此启动脚本不再优先查找 `.python-core`，也不再偷偷回退系统 Python，而是直接检查本机 Python 版本，不满足条件时阻断并给出提示。README 和安装器说明也同步说明这是源码分发包，需要 Python 3.10 或更高版本。

4. FastAPI 安全边界补齐

   重写 `api.py` 的基础安全边界：

   - 默认建议只监听 `127.0.0.1`，不鼓励公网暴露。
   - 支持通过 `AI_OPS_API_TOKEN` 配置 Bearer Token。
   - 配置 Token 后，所有业务接口必须带 `Authorization: Bearer <token>`。
   - 未配置 Token 时，明确属于“本机开发模式”。
   - API 默认不持久化任何生成结果。
   - 只有同时满足 `AI_OPS_API_PERSISTENCE=1` 和请求体 `persist: true` 时，才写入数据库。

   这避免了“接口能被随便调用、内容静默入库”的风险。

5. 本地持久化改为显式保存

   将个人问答、用户洞察、版本分析、竞品分析、协作讨论、活动方案等输出从“生成后自动保存”调整为“页面展示后由用户点击保存”。

   用户保存前可以看到保存位置和 JSON 附加文件状态。系统新增两个统一开关：

   - `AI_OPS_ENABLE_LOCAL_SAVE=0`：彻底关闭本地报告保存。
   - `AI_OPS_SAVE_JSON_SIDECAR=0`：保存 Markdown 时不生成 JSON 附加文件。

   这让用户清楚知道哪些内容会落盘、落到哪里、以什么格式保存。

6. 管理后台补充数据保留策略

   管理后台新增数据保留说明，展示用户数据目录、本地保存开关和 JSON 附加文件状态，并提供本地报告与活动方案清理入口。

7. 数据库默认位置迁移

   `database.py` 的默认 SQLite 数据库从项目根目录迁移到用户数据目录。这样即使未来安装到 `Program Files` 这类只读目录，数据库写入也不会失败。

8. 知识库索引默认位置迁移

   `build_index.py` 的 `chroma_db` 位置迁移到用户数据目录。管理后台的知识库状态也同步读取用户数据目录下的索引，避免显示和实际运行路径不一致。

9. 自检脚本强化

   `health_check.py` 增加运行时目录测试环境，避免自检时因为系统默认数据目录权限问题失败。同时扩展临时目录排除规则，避免历史临时运行目录干扰编码扫描。

10. 文档与分发说明同步

   README、`.env.example`、`setup.iss` 和源码包内容同步更新，明确说明当前包是源代码分发版、运行时数据写入用户数据目录、后台必须初始化密码、API 默认仅适合本机演示、API 持久化默认关闭、本地报告默认不自动保存。

### 解决的问题

V15 主要解决五类高优先级问题：

- 安装目录与运行时写入冲突：运行时文件统一进入用户数据目录。
- 后台公开口令风险：首次使用必须初始化密码，且只保存哈希。
- 安装链路表达不一致：明确要求 Python 3.10+，不伪装成独立运行包。
- API 未授权调用和静默入库风险：支持 Bearer Token，默认不持久化结果。
- 用户内容默认明文落盘风险：所有报告类结果都由用户显式点击保存。

### 能力证明

V15 证明这个项目不只是“能跑的作品集”，还开始具备真实产品交付意识。它体现了我对运行环境、安全边界、隐私策略、数据保留和交付可解释性的理解。

### 验证结果

V15 完成后进行了以下验证：

- 核心文件通过 Python 语法检查。
- `health_check.py` 全部通过。
- 无 API Key 时仍可进入演示模式。
- 后台未初始化密码时不会使用公开口令放行。

## V17：配置中心化、后端预接口与专业研究工具补全

### 迭代背景

V17 是对前几轮密集更新的一次内容补账和架构归档。V15 与 V16 重点记录了安全、分发、运行时目录和显式保存等高优先级修正，但项目中还有几类已经完成的能力升级没有被完整写进迭代记录：配置中心化、业务逻辑分层、FastAPI 后端预接口、数据库预接入、安装向导、以及访谈助手、问卷工坊、竞品雷达的专业化升级。

这些内容不只是“又加了几个文件”，而是让项目从一个 Streamlit 单体应用，进一步向“可配置、可扩展、可接入后端、可被技术同学审查”的作品集系统靠近。

### 关键改动

1. 配置中心化继续落地

   项目新增并完善 `config/` 配置体系，将运行时可配置内容从代码和旧文本提示词中逐步抽离出来。

   主要配置包括：

   - `config/app.yaml`：应用名称、模型参数、文件夹路径等全局配置。
   - `config/agents/*.yaml`：多职能协作讨论中的十个运营角色配置。
   - `config/prompts/modules/*.yaml`：个人背景、作品集、实习经历、项目经历、游戏理解、提示词工程等模块化背景配置。
   - `config/prompts/tasks/*.yaml`：版本分析、竞品雷达、活动策划、反馈清洗、访谈助手、问卷工坊、视频舆情等任务型提示词配置。

   同时新增 `config_loader.py` 作为配置翻译层。它负责读取 YAML 配置，并将结构化字段翻译成大模型能理解的自然语言系统提示词。这样后续修改角色职责、分析框架、输出风格和任务规则时，可以优先改 YAML，而不是直接改 `app.py`。

2. YAML 成为运行时主要数据源

   原有的 `*_prompt.txt` 文件被保留为人类可读的历史文档和存档，但运行时提示词加载逻辑改为优先读取 `config/` 下的 YAML 文件。

   如果 YAML 文件缺失或格式异常，系统回退到代码内置的最小默认提示词，而不是再回读旧 txt 文件。这让运行时数据源更明确，避免出现“同一条规则在 txt、YAML、代码中多处存在，最后不知道谁生效”的问题。

3. `core.py` 与 `Orchestrator` 分层重构

   为了降低 `app.py` 的复杂度，项目新增 `core.py`，将模型调用、RAG 查询、LangGraph 调用和多职能协作讨论等核心业务逻辑封装进 `Orchestrator` 类。

   典型方法包括：

   - `run_chat()`：处理“了解我”聊天。
   - `run_version_analysis()`：处理版本分析。
   - `run_feedback_clean()`：处理反馈清洗。
   - `run_collaboration()`：处理多职能协作讨论。

   这个改动让 `app.py` 更接近展示层，`core.py` 更接近业务编排层。后续如果增加 FastAPI、命令行工具或其他前端入口，可以复用同一套核心逻辑，而不是在多个界面里重复写 API 调用代码。

4. FastAPI 后端预接口

   项目新增 `api.py`，为未来接入公司内部系统或前后端分离做预留。

   当前接口包括：

   - `GET /health`：健康检查。
   - `POST /chat`：聊天问答。
   - `POST /version-analysis`：版本分析。
   - `POST /feedback-clean`：反馈清洗。
   - `POST /collaboration`：多职能协作讨论。
   - `POST /competitor-analysis`：竞品分析。
   - `POST /activity-workshop`：活动策划。
   - `POST /rag-query`：RAG 查询。

   FastAPI 层调用的是 `core.Orchestrator`，而不是重新实现一套生成逻辑。这说明项目已经开始具备“一个核心，多种入口”的架构意识。

5. 数据库预接入设计

   项目新增 `database.py`，使用 SQLAlchemy 定义基础数据模型，为未来接入公司统一数据库做准备。

   主要模型包括：

   - `User`：记录用户 ID 和用户名。
   - `AnalysisReport`：记录报告 ID、创建者、报告类型、Markdown 内容、JSON 数据和创建时间。

   当前默认数据库为用户数据目录下的本地 SQLite 文件，并通过 `DATABASE_URL` 环境变量支持未来替换成公司数据库。API 写入默认关闭，只有在显式开启持久化并由请求声明 `persist: true` 时才写入，避免静默保存敏感内容。

6. 安装向导与源码分发链路

   项目新增 `install.py`，用于引导用户完成环境检查、依赖安装、API Key 配置、后台密码初始化、知识库索引构建和启动应用。

   同时补充：

   - `installer.spec`：PyInstaller 打包入口配置。
   - `setup.iss`：Windows 安装器脚本草案。
   - `run.bat` / `run.sh` / `start_app.bat`：本地启动脚本。
   - README 快速开始与分发说明。

   后续审查中确认当前定位为“源码分发版”，因此启动脚本不再假装自带 Python 运行时，而是明确要求 Python 3.10+。这让分发方式更诚实，也降低普通用户在无 Python 环境下误以为安装包能独立运行的风险。

7. 访谈助手升级为专业用户研究工具

   访谈助手不再只是静态提纲生成器，而是在 YAML 配置中加入更完整的用户研究方法论。

   新增能力包括：

   - 五层用户画像框架：玩家与产品的情感关系、娱乐形式态度、玩法偏好、人口统计变量、行为量化指标。
   - 追问预案系统：针对正面详细、正面简略、负面详细、负面简略、模糊回答、反问、拒绝回答、情绪化回答等情况给出具体追问话术。
   - 访谈观察记录表：记录情绪波动节点、回避行为、主动展开话题、高频词、自创词和身体语言备注。

   这让访谈助手输出的不只是“问题列表”，而是一套能执行、能观察、能复盘的访谈工具。

8. 问卷工坊升级为量化研究工具

   问卷工坊在 YAML 配置中加入了更完整的量化研究规则。

   新增能力包括：

   - 样本质量控制：语义矛盾对、量表趋同检验、无效样本剔除标准。
   - 信效度设计：每个心理维度至少三道题，并在分析框架中说明维度和题号关系。
   - 反向计分：生成投放版和分析师版，分析师版标注反向题和计分说明。
   - 预测试机制：30-50 份预测试样本、填写时长、内矛盾率、量表趋同、Cronbach's alpha、天花板/地板效应、开放题填答率等检查项。

   这让问卷工坊从“生成题目”升级为“设计、投放、质控、分析”的完整调研工具链。

9. 竞品雷达升级为专业策略分析系统

   竞品雷达在 YAML 配置中加入了更强的策略分析框架。

   新增能力包括：

   - 竞品分类框架：同玩法同生态位、同生态位不同玩法、同玩法不同生态位三类竞品，并根据类别调整分析权重。
   - 拆解、模仿、超越：每个可借鉴点必须说明竞品如何实现、我方如何适配、以及如何做出差异化。
   - 2x2 优先级矩阵：按预期效果和实施成本划分速赢机会、长期规划项、低优先级可选项、高风险陷阱。
   - 高风险陷阱分析：重点识别隐藏成本、品牌同质化、用户心智定位偏移。
   - 借鉴风险自查：检查玩家感知风险、品牌调性风险、数据适配风险。
   - 版本趋势分析：当用户提供竞品连续两个版本公告时，分析活动力度、付费节奏和宣发策略变化。

   这让竞品雷达不再只是“竞品做了什么”的静态汇总，而是能把竞品动态转化为可执行、可放弃、可讨论的策略选项。

   这份报告说明 SEO 提示词工程和 AI 运营分身 Agent 不是两个孤立项目，而是在底层方法论上共享同一组发现：

   - 指令加载时序
   - 结构化分工

   - 多维并行校验

### 解决的问题

V17 主要解决五类问题：

- 配置散落问题：运行时规则从代码、txt、YAML 混杂，逐步收束到 YAML 配置体系。
- 单体复杂度问题：`app.py` 承担过多业务逻辑，通过 `core.Orchestrator` 拆出核心编排层。
- 后续接入问题：FastAPI 和数据库模型为未来接入公司内部系统预留接口。
- 专业深度问题：访谈、问卷、竞品三个工具从通用生成升级为更贴近真实运营研究和策略工作的专业框架。
- 作品集方法论问题：通过协同演进报告，把提示词工程和 Agent 架构之间的底层关系讲清楚。

### 能力证明

V17 证明这个 Agent 已经不只是一个网页 Demo，而是开始具备三层能力：

1. 配置化能力  
   能把经常变化的角色设定、分析框架、输出规则和任务提示词外置，让非核心代码也能被持续维护。

2. 架构分层能力  
   能把前端展示、业务编排、模型调用、RAG 检索、API 服务和数据库预留拆成更清晰的层次。

3. 运营方法论沉淀能力  
   能把访谈、问卷、竞品分析这些真实运营工作拆成可复用框架，而不是只依赖一句“大模型帮我生成”。
