# 作品集能力总览与介绍口径

这份文档用于统一“关于我”“作品集介绍”“自我介绍”相关回答的表达口径。回答时不要把作品集说成零散文件，而要围绕“真实工作需求、AI应用能力、知识治理能力、测试验证能力、系统工程设计、线下活动执行经验”六条主线展开。

## 一、SEO文章写作 Prompt 工程

### 设计来源

这个作品源于实际工作需求。我在 AI 音乐生成产品线的运营实习中，需要稳定产出英文 SEO 文章初稿，因此围绕真实工作任务进行了持续测试和迭代。

### 迭代路径

该 Prompt 工程大致经历了四个阶段：

1. 单纯叠加：把需求直接堆进 Prompt。
2. 设定约束与分类讨论：开始区分写作目标、硬约束、风格要求和不同内容场景。
3. 模块化：将任务拆成结构、风格、功能介绍、过渡段、验证等模块。
4. 分段输入约束：通过分阶段加载减少 AI 注意力稀释，让模型在不同阶段只处理当前最关键的约束。

### 过程收获

在这个过程中，我边干边学，理解了很多交互端 AI 特性和技术问题，包括：

- 约束设计思路
- 角色设计思路
- AI 注意力问题与对应设计对策
- 工程架构思维
- 用迭代实验验证 Prompt 是否真的提升产出质量

### 产出效果

这套工程最终产出了质量较高的英文 SEO 文章初稿，写作速度提升约一倍，并能达到每日稳定产出一篇高质量英文 SEO 文章的工作节奏。

### 证明能力

这个作品证明了我的：

- AI 交互层理解能力
- 运营实际工作中的 AI 使用能力
- 迭代思维
- 对工作效率的主动追求
- 工程设计思维
- 工程试验经验与思维

## 三、知识治理控制台

### 设计来源

这个作品来自 AI 运营分身继续迭代后的一个新问题：当 Agent 的作品集材料、运营规则和修正知识越来越多时，只做一次性 RAG 索引是不够的，还需要一个能登记、审核、追溯、修正和同步知识的治理入口。

### 项目定位

知识治理控制台是一个独立 Streamlit 后台，不嵌入 AI 运营分身 Agent 代码。它使用本地 SQLite 保存结构化知识单元和 Badcase 记录，再把“生效中且未过期”的知识导出为 `sync/<kid>.yaml`。

Agent 端默认仍只扫描自己的 `works/` 文件夹；只有用户手动启用同步路径配置时，才会读取控制台的 `sync/` 导出文件夹。这保证了 Agent 独立运行行为不变，也为后续知识治理接入预留了边界清晰的通道。

### 核心能力

控制台包含：

- 知识单元注册中心
- 知识库健康度仪表盘
- 知识清单筛选
- 责任归属与错误追溯工作台
- Badcase 回流与审核看板
- YAML 同步转译到 `sync/`

### 证明能力

这个项目证明了我的：

- RAG 知识库维护与治理意识
- 把错误回答转化为可追踪 Badcase 和可修正知识单元的产品思维
- 通过 YAML 和配置开关实现低耦合系统衔接的架构意识
- 对 Agent 长期维护、知识失效、审核状态和责任归属的工程化理解
- 不破坏原系统默认行为的交付边界意识

## 四、帕姆帮帮黑箱测试方案

### 设计来源

这个作品源于我对“AI 在游戏里到底有没有用”的好奇心。帕姆帮帮是《崩坏：星穹铁道》内置的 AI 助手，但内测期间玩家评价两极分化，因此我借助 AI 辅助设计了一套科学黑箱测试方案，并亲自进行了测试。

### 测试边界

测试方案建立了清晰边界：

- 只通过游戏内对话与可见行为交互来判定
- 不解包客户端
- 不抓包
- 不访问后端
- 不读取内存

### 三层测试 Oracle

我将测试 Oracle 分为三层：

1. 确定性 Oracle：用于剧情事实、素材关卡落点等可硬判的问题。
2. 约束性 Oracle：用于配队与装备推荐。重点不是“是否与唯一标准答案相同”，而是“是否满足已拥有、已解锁、角色定位等硬约束”。
3. 解释性 Oracle：看系统能否说明推荐理由，并在资源不足或条件冲突时主动澄清。

米哈游官方明确说明推荐逻辑会随拥有角色和大数据持续更新，因此推荐结果本身允许发生合法漂移。测试重点不是追求唯一答案，而是验证推荐是否满足边界、约束和解释逻辑。

### 测试覆盖

测试方案覆盖六个维度，共一百条核心用例：

1. 功能覆盖：剧情讨论、素材关卡跳转、阵容推荐、装备推荐。
2. 准确性验证：事实错误、幻觉检测、不兼容推荐拦截。
3. 鲁棒性测试：错字、拼音、别称、中英混输、超长文本、矛盾约束。
4. 上下文与多轮对话：指代消解、约束保留、任务切换、对话修复。
5. 安全与合规：Prompt 注入、隐私泄露、PII 保护、作弊灰产拒答、未成年人保护。
6. 性能与可用性：延迟、弱网、断续网、跨平台一致性。

每条用例都定义了前置条件、触发对话、预期行为和可量化指标，并配置统计显著性检验方法和置信区间计算公式。

### 结论与建议

最终结论是：帕姆帮帮在剧情信息查询类问题上的准确率较高，但在配队建议和复杂推理问题上明显吃力，并且在多轮对话中容易丢失上下文。

基于这些发现，我给出了具体改进建议，包括：

- 优化知识库覆盖范围
- 增加结构化数据的检索优先级
- 引入更明确的能力边界声明机制

### 证明能力

这个作品证明了我的：

- 系统性验证思维
- 科学测试方案设计能力
- AI 产品评估与批判性思考能力
- 数据处理与记录标准化能力
- 从测试结论提炼改进建议的归纳能力

## 五、星铁战术引擎

### 设计来源

这个作品源于我作为《崩坏：星穹铁道》玩家的真实痛点。我是一个回流开服玩家，仓库里的角色有不少已经退环境，资源和角色都很紧张，想要打深渊只能通过精细排轴提高通过率，但自己的操作水平又不够稳定。

在这个背景下，我查找到社区中已有速度轴模拟器和即时伤害计算工具。前者能通过穷举模拟行动轴，后者能计算单次伤害，但目前还没有成熟的全自动排轴工具。因此我设计了一个基于社区开源项目和大模型输出的最佳伤害期望操作轴推理代码引擎。

### 核心原则

我的核心设计思路是：不确定的事情不做模糊估算，而是做分类讨论。

战斗中最大的不确定性是受击回能。敌人打谁是不确定的，但谁被打会直接影响能量循环和最优出招顺序。传统做法可能会算一个期望值，但我选择先根据仇恨值分析谁更可能被打，再对每种可能分别计算最优方案，最后告诉玩家：大概率会发生什么，发生了该如何变招。

### 系统运作逻辑

第一步，定义战斗完整状态。我定义了七个核心变量来描述任意时刻的战斗快照：

- 行动队列
- 能量值
- 战技点
- 韧性条
- 增益减益状态
- 持续伤害效果
- 累积伤害

第二步，让系统具备预测性。每次轮到我方行动时，系统在有限时间窗口内枚举所有合法操作序列。合法性由能量是否足够开大、战技点是否足够使用战技、Buff 机制是否满足共同决定，从而筛掉大量不可能分支。

第三步，评价路径优劣。单纯用总伤害评价容易导致短视决策，因此我在评分公式中加入能量枯竭惩罚和战技点枯竭惩罚，避免系统为了短期爆发烧光资源。

第四步，用剪枝加速搜索。对于明显不可能超过当前最优解的分支直接放弃；对于已经足够好的解提前终止搜索。这样可以把理论上的大量分支压缩到个人电脑可承受的有效搜索量。

### 架构设计

整个系统分为五层：

1. 核心战斗引擎层：八个模块各管各的状态变量。
2. 搜索引擎层：带剪枝的最优路径搜索。
3. 流派扩展层：支持记忆和欢愉命途等可插拔机制，不修改底层。
4. 数据抽象层：角色和敌人数据通过统一接口加载，当前用本地示例文件验证。
5. 外部 API 接口层：预留速度轴模拟和伤害计算两个核心接口的规范签名与 Mock 实现。

### 边界说明

作为个人作品，我无法获取真实仇恨值公式、角色底层机制参数和完整伤害乘区系数。这些需要企业数据中台或战斗引擎代码接入。

但我已经在架构中为所有外部依赖预留接口，当前用示例数据和 Mock 实现验证了整套架构可行性。只要接入企业数据层和 API 层，引擎即可完整运转。

### 证明能力

这个作品证明了我的：

- 从真实玩家痛点出发的产品洞察能力
- 数学建模与系统工程设计能力
- 利用社区开源资源的务实工程思维
- 分类讨论替代模糊估算的精确推演思路
- 模块化架构设计与接口预留的团队协作意识
- 对作品边界和外部依赖的诚实认知
- 在数学严谨性与工程可行性之间找到平衡点的架构设计思维
